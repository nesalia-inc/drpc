# TypeScript 5.7 Breaking Changes & Migration

Date: 2026-04-21
Tag: [typescript] [v5.7] [breaking-changes] [migration]

## Overview

TypeScript 5.7 brings several quality-of-life improvements including checks for never-initialized variables, relative path rewriting, and ES2024 support.

## New Features

### 1. Checks for Never-Initialized Variables

The type system now reports errors when variables have "never" been initialized at all, even when accessed in separate functions.

### 2. Path Rewriting for Relative Paths

New compiler option `--rewriteRelativeImportExtensions` rewrites `.ts` imports to `.js` in output:

```typescript
// Input
import { foo } from "./foo.ts";

// Output
import { foo } from "./foo.js";
```

### 3. ES2024 Support

Added `--target es2024` and `--lib es2024`, enabling:
- `Object.groupBy`
- `Map.groupBy`
- `Promise.withResolvers`

### 4. Ancestor Configuration File Search

Editor now walks up the directory tree to find the most appropriate `tsconfig.json` instead of stopping at the first match.

### 5. Faster Composite Project Checks

TypeScript 5.7 only checks if a file belongs to the root set of files in composite projects, avoiding extreme slowdowns.

### 6. Validated JSON Imports

Under `--module nodenext`, JSON imports now require `type: "json"` import attribute and use only default exports.

### 7. V8 Compile Caching

Leverage Node.js 22's `module.enableCompileCache()` for ~2.5x speed-up in `tsc --version`.

## Breaking Changes

### 1. TypedArrays Now Generic (HIGH IMPACT)

`TypedArray`s are now generic over `ArrayBufferLike`:

```typescript
// Old
Uint8Array

// New
Uint8Array<ArrayBufferLike> // or Uint8Array<ArrayBuffer>
```

This may cause errors when passing TypedArrays to functions expecting `BufferSource`.

### 2. Non-literal Method Names in Classes

Non-literal method names in classes now create index signatures, changing how classes with symbol-computed properties are typed.

### 3. More Implicit `any` Errors

Functions returning `null`/`undefined` outside `strictNullChecks` now produce more implicit `any` errors.

### 4. `lib.d.ts` DOM Types

DOM type changes may introduce new type-checking errors.

## Migration Notes

### Fix TypedArray Errors

1. **Update @types/node**:
   ```bash
   npm update @types/node --save-dev
   ```

2. **Pass `.buffer` property**:
   ```typescript
   // Instead of:
   someFunc(data) // data is TypedArray

   // Use:
   someFunc(data.buffer as ArrayBuffer)
   ```

3. **Use explicit type parameters**:
   ```typescript
   const arr: Uint8Array<ArrayBuffer> = new Uint8Array(data);
   ```

## Impact Assessment

| Area | Impact |
|------|--------|
| TypedArray generics | High |
| Implicit any | Medium |
| DOM types | Low |
