See how DeesseJS RPC stacks up against other solutions in terms of type safety, performance, and developer experience.

## Overview

Choosing the right communication protocol between your client and server is one of the foundational architectural decisions you will make. The choice affects how quickly you can iterate, how safely you can make changes, and how your team scales as you add features. DeesseJS RPC provides an opinionated approach that prioritizes type safety, simplicity, and a great developer experience without requiring you to learn a new query language or commit to a specific frontend framework.

This comparison walks through how DeesseJS RPC measures against three popular alternatives: tRPC, REST, and GraphQL. Each section covers type safety, API design, and performance so you can understand the trade-offs involved.

## DeesseJS RPC vs tRPC

### Type Safety

Both DeesseJS RPC and tRPC share a core philosophy of end-to-end type safety, where the TypeScript types you define on the server propagate automatically to the client. This eliminates an entire class of bugs where the client and server drift out of sync. However, the two solutions take different approaches to how that type information is exposed and used.

When you use tRPC, you typically work with React Query hooks that carry the type information through the hook system. The type inference flows through the generic parameters of hooks like `useQuery` and `useMutation`. DeesseJS also provides React hooks, but it ships a vanilla JavaScript client as the primary offering. This means you can use DeesseJS RPC in any JavaScript environment, not just React applications. The type inference works through procedure calls on the client, and those types are inferred directly from the server router definition.

The error handling models differ as well. DeesseJS RPC uses an explicit Result type with `ok` and `err` properties, which makes every response self-describing. You always know whether an operation succeeded or failed by inspecting the discriminant property. In contrast, tRPC relies on throwing exceptions to signal errors. When a tRPC handler throws, the error propagates through the React Query error state. This approach works well in React contexts but feels less predictable when you are writing non-React code or server-to-server communication.

The example below shows how type safety looks on the client when using DeesseJS RPC. You define your router once on the server, and the types flow through to the client call without any additional type annotations.

```typescript
// Server: api.ts
import { defineContext } from '@deessejs/server'
import { z } from 'zod'
import { ok, err } from '@deessejs/fp'
import { NotFoundException } from '@deessejs/server'

const { t, createAPI } = defineContext({
  context: () => ({ db: myDatabase }),
})

const getUser = t.query({
  args: z.object({ id: z.number() }),
  handler: async (ctx, args) => {
    const user = await ctx.db.users.find(args.id)
    if (!user) {
      return err(new NotFoundException('User not found'))
    }
    return ok(user)
  },
})

export const appRouter = t.router({ users: t.router({ get: getUser }) })
export type AppRouter = typeof appRouter
```

```typescript
// Client: userService.ts
import { createClient } from '@deessejs/client'
import type { AppRouter } from './api'

const client = createClient<AppRouter>('https://api.example.com')

// Types are fully inferred from the server router
const result = await client.users.get({ id: 42 })

if (result.ok) {
  console.log(result.value) // User type is fully inferred
} else {
  console.error(result.error) // Error type is also inferred
}
```

### API Design

The API design philosophies diverge in important ways. tRPC is deeply tied to React and expects you to build your client around React Query hooks. The queries and mutations live within that hook paradigm, which works well for React applications but creates friction when you need to call your API from non-React contexts like background jobs, command-line tools, or other servers.

DeesseJS RPC separates the transport from the API definition more cleanly. You get a vanilla JavaScript client that can call procedures just like regular async functions. If you are building a React application, you can use the React hooks wrapper, but it is optional. This means the same API works equally well for a Next.js app, a vanilla TypeScript script, a Node.js microservice, or an Electron desktop application.

Both frameworks let you organize procedures into namespaces and routers, which helps keep related operations grouped together. DeesseJS adds a plugin system that lets you extend the context available to every handler with reusable functionality. You can also attach middleware globally or per-procedure to handle cross-cutting concerns like authentication and rate limiting.

The lifecycle hooks in DeesseJS RPC give you points to intercept before and after procedure execution. You can hook into `beforeInvoke`, `afterInvoke`, `onSuccess`, and `onError` to add logging, monitoring, or modify behavior without touching the handler logic itself.

### Performance

In terms of raw performance, both frameworks are competitive because they both send JSON over HTTP and use similar underlying mechanisms. The performance differences come from how each framework handles serialization, validation, and the overhead of the type inference system at runtime.

DeesseJS RPC uses Zod for input validation, which means every argument goes through schema parsing before reaching your handler. This adds a small overhead per request, but it guarantees that your handler receives well-typed data. The validation cost is typically measured in microseconds and is dwarfed by database or network latency in most applications.

tRPC performs validation differently and relies on the underlying HTTP framework for some of that work. The performance characteristics are similar in practice, and both frameworks are fast enough that the choice rarely comes down to raw throughput.

The areas where performance might influence your decision are around bundle size and client-side overhead. DeesseJS ships a smaller client footprint since it is not coupled to React Query, which itself is a large library. If bundle size is a critical constraint, DeesseJS may have an edge, though both are reasonable choices for production applications.

## DeesseJS RPC vs REST

### Type Safety

REST as an architectural style says nothing about types. When you build a REST API, the server defines contracts through documentation, and the client implements against those contracts. If the server changes a response shape and the client is not updated, you get runtime errors that TypeScript cannot catch.

DeesseJS RPC brings full type safety to the equation. The types you define on the server are inferred on the client through the client creation process. When you change a procedure on the server, TypeScript will flag any client code that no longer matches the new shape. This compile-time safety means refactoring is safer and faster because the compiler tells you about mismatches before you run the code.

REST APIs typically require an additional step like generating TypeScript types from an OpenAPI spec or using a tool like tRPC codegen to get similar safety. Even when you generate types from OpenAPI, the generated types are only as good as the spec itself, and keeping the spec in sync with implementation is a discipline problem, not a technical one.

With DeesseJS, the type inference happens directly from the implementation without any generation step. When you write the handler, the types are immediately available to the client. No schema to update, no types to regenerate, no documentation to keep in sync.

The Result type in DeesseJS also provides better error safety than REST. In REST, you might return a 404 status code, and the client has to interpret that status to know what happened. The body of the response might be a JSON object or might be empty depending on the server implementation. In DeesseJS, every response has a consistent shape with `ok` and `err` discriminants, and the error type is part of the contract.

```typescript
// REST: client-side, no type safety
const response = await fetch('/api/users/42')
// What is the shape of response.json()? TypeScript does not know.
// Was it a 404? A 500? You have to check and handle all cases manually.

if (response.ok) {
  const data = await response.json() // any type
  console.log(data.name) // Might be undefined if API changed
} else {
  // Handle error case
}
```

```typescript
// DeesseJS RPC: client-side, full type safety
const result = await client.users.get({ id: 42 })

// TypeScript knows the exact shape of both success and error cases
if (result.ok) {
  console.log(result.value.name) // Guaranteed to exist if ok is true
} else {
  console.error(result.error.message) // Error type is inferred
}
```

### API Design

REST is built around HTTP resources and HTTP verbs. You design around nouns, and the verbs are limited to GET, POST, PUT, DELETE, PATCH, and a few others. This works well for CRUD operations but becomes awkward when you need to model actions that do not fit neatly into those categories.

DeesseJS RPC is procedure-based. You define named procedures that can perform any operation, not just create, read, update, or delete a resource. A procedure might trigger a workflow, send an email, process a batch job, or perform a complex calculation. The procedure name is part of the API surface, which makes intent explicit.

The REST approach of embedding pagination parameters in query strings and response metadata in headers is conventions-based. Different APIs implement pagination differently, and the client has to learn each convention. DeesseJS procedures return typed results directly, and pagination becomes just another part of the return value.

DeesseJS also provides automatic cache key management. When you return a result from a query, you include cache keys that the client uses to store and invalidate the cached data. REST has no built-in caching model; you can use HTTP cache headers, but implementing cache invalidation logic is left entirely to you.

The example below shows how you might model a complex operation in REST versus DeesseJS RPC. The REST approach would typically require multiple endpoints or a generic "action" endpoint with a request body that varies.

```typescript
// REST: POST /api/users/42/complex-action
// Request body varies depending on action
fetch('/api/users/42/complex-action', {
  method: 'POST',
  body: JSON.stringify({ action: 'process_payment', amount: 100 })
})
```

```typescript
// DeesseJS RPC: t.mutation() with a clear name
const processPayment = t.mutation({
  args: z.object({ userId: z.number(), amount: z.number() }),
  handler: async (ctx, args) => {
    const result = await paymentService.charge(args.userId, args.amount)
    return ok(result)
  },
})
```

### Performance

REST and DeesseJS RPC both use HTTP as the transport, so the baseline network performance is similar. The key performance differences come from how each handles serialization and how much overhead the framework introduces.

REST APIs typically return raw JSON responses. When you use a database client inside a REST handler, the result goes directly into the HTTP response. DeesseJS wraps results in the Result type, which adds a thin layer around the response. This overhead is minimal in practice, typically a few bytes per response.

The cache system in DeesseJS can actually improve perceived performance by reducing redundant server calls. When the client has a valid cached result, it returns immediately without a network request. REST APIs require you to implement your own caching logic, which is often an afterthought.

On the server side, DeesseJS procedures go through middleware and lifecycle hooks that add a small amount of overhead per call. For most applications, this overhead is negligible compared to the actual handler logic and database operations. If you are building a high-throughput system where every microsecond matters, you would need to benchmark your specific workload.

## DeesseJS RPC vs GraphQL

### Type Safety

GraphQL has a famous feature called runtime introspection. At runtime, a GraphQL client can query the schema to discover what types, queries, and mutations are available. This enables powerful developer tools like GraphiQL and Apollo Studio that let you explore the API interactively.

DeesseJS RPC takes the opposite approach. Type safety is enforced at compile time through TypeScript inference. When you write the server router, the types are extracted and used to create the client. If the types do not match, TypeScript tells you at compile time. There is no runtime introspection because there is no need for it; the types are verified before the code runs.

The advantage of compile-time safety is that you catch mismatches earlier in development, often before you even run the code. GraphQL's runtime introspection means a client can ask for fields that no longer exist, and the server returns an error at runtime. With DeesseJS, a client that asks for a removed field would be a TypeScript compilation error.

GraphQL also requires you to define a schema language that is separate from your implementation. You might define a GraphQL schema in SDL and then write resolvers that implement it. Over time, the schema and resolvers can drift, and nothing enforces that they stay in sync. In DeesseJS, the handler IS the type definition. There is no separate schema that can get out of sync.

The error handling story is similar to REST in that GraphQL errors are returned alongside successful data in the same response. DeesseJS uses the Result type to make success and failure paths explicit and forces the caller to handle both.

```typescript
// GraphQL: separate schema from implementation
// schema.graphql
// type Query { user(id: ID!): User }
// type Mutation { updateUser(id: ID!, name: String!): User }

// resolver.ts
// const resolvers = {
//   Query: { user: (_, { id }) => db.users.find(id) },
//   Mutation: { updateUser: (_, { id, name }) => db.users.update(id, { name }) }
// }

// Client has no compile-time guarantee that resolvers match schema
```

```typescript
// DeesseJS RPC: single implementation, types inferred
const { t, createAPI } = defineContext({
  context: () => ({ db: myDatabase }),
})

const getUser = t.query({
  args: z.object({ id: z.number() }),
  handler: async (ctx, args) => {
    const user = await ctx.db.users.find(args.id)
    if (!user) {
      return err(new Error('User not found'))
    }
    return ok(user)
  },
})

// Client types are directly inferred from this implementation
// No schema drift possible
```

### API Design

GraphQL ships with its own query language. When a client wants to fetch data, it writes a query string that asks for exactly the fields it needs. This is powerful because the client has fine-grained control over the response shape.

DeesseJS RPC uses simple function calls instead. To fetch a user, you call `client.users.get({ id: 42 })`. The response shape is whatever the handler returns, and the client cannot customize it on the fly. This is less flexible but also simpler to understand and work with.

The GraphQL query language adds overhead in several places. The client must construct a query string, send it to the server, and the server must parse and validate that query string against the schema. DeesseJS RPC sends a simple JSON payload with the procedure name and arguments. There is no query parsing step, no validation against a schema at runtime, and no complex execution engine.

For applications where clients have very different data needs and you want to let each client fetch exactly what it needs, GraphQL's query language is a genuine advantage. For applications where the server knows the best way to expose data and you prefer simplicity, DeesseJS RPC is easier to work with.

Subscriptions are another area where the approaches differ. GraphQL has a built-in subscription mechanism through WebSocket connections. DeesseJS uses an event system that integrates with the same procedure model. When a mutation occurs, you can emit events that clients can subscribe to for real-time updates. The DeesseJS approach keeps everything within the same mental model rather than learning a separate subscription syntax.

```typescript
// GraphQL: learning the query language
const query = `
  query GetUser($id: ID!) {
    user(id: $id) {
      id
      name
      email
      posts {
        title
      }
    }
  }
`

// Must parse and validate query at runtime
// Different from normal function calls
```

```typescript
// DeesseJS RPC: simple function call
const result = await client.users.get({ id: 42 })

// If you need related data, the handler includes it or you call another procedure
// Same as calling a regular async function
```

### Performance

GraphQL's flexible query language comes with performance costs. When a client sends a query like `{ user { posts { comments { author } } } }`, the server must parse the query, validate it against the schema, and then execute a resolver for each field in the selection set. This can lead to N+1 query problems where a list of users triggers one query per user to fetch their posts.

DeesseJS procedures are simpler. The client calls a procedure, the server executes the handler, and returns the result. There is no nested resolver execution, no query parsing, and no risk of accidentally triggering hundreds of database queries from a single client request. If you need to fetch related data in a DeesseJS procedure, you do so explicitly in the handler, and you can optimize that fetch however you like.

The lack of a query language also means DeesseJS responses tend to be smaller because there is no metadata overhead from the GraphQL response format. When you return `{ ok: true, value: user }`, the response is exactly what the handler produced with no additional introspection or type metadata.

For applications serving many different client types with varied data requirements, GraphQL's flexibility can justify the overhead. For applications with more predictable data access patterns, DeesseJS RPC keeps things lean and fast.

## When to Choose DeesseJS RPC

DeesseJS RPC is the best choice when you want end-to-end type safety without the overhead of a query language or the complexity of a schema definition step. If your team is working primarily in TypeScript and you want the client and server to stay in sync through the type system alone, DeesseJS gives you that without additional tooling.

Choose DeesseJS RPC when you need to call your API from multiple environments. The vanilla JavaScript client works in browsers, Node.js, Deno, and other JavaScript runtimes. If you are building a web app alongside a CLI tool, a mobile app, or a microservices architecture, the same API definition serves all of them without framework coupling.

When your application benefits from the plugin system, middleware, and lifecycle hooks, DeesseJS provides those as first-class features. The plugin system lets you encapsulate reusable functionality and share it across procedures without copy-pasting code.

If you prefer explicit error handling over exceptions, the Result type enforces that pattern naturally. When you want consistent caching behavior with automatic key management, DeesseJS has that built in rather than requiring you to implement it from scratch.

Consider alternatives when your clients need highly variable response shapes at runtime, in which case GraphQL's query language provides more flexibility. When you need deep integration with the React Query ecosystem and your application is React-only, tRPC offers a tight integration that some teams prefer. When you are building public APIs that third parties will call with arbitrary HTTP clients, REST's universal compatibility may be more important than type safety.

For teams that value simplicity, type safety, and a framework-agnostic approach, DeesseJS RPC provides a solid foundation for building client-server communication that scales from simple prototypes to production applications.
